# Workspace

This is the root workspace. It contains your projects (code) and notes (knowledge) side by side. Your agent reads this file to understand how things are organized.

---

## Structure

```
workspace/
├── CLAUDE.md              # this file — workspace instructions
├── projects/              # code lives here, each project has its own repo
│   └── {project-name}/
│       ├── CLAUDE.md      # project-specific agent instructions
│       ├── ledger.md      # session handoff — what happened, what's next
│       ├── README.md
│       └── docs/
│           ├── plan.md
│           └── resources.md
├── notes/
│   └── vault/             # markdown vault (Obsidian or any markdown directory)
│       ├── tasks.md        # master task list
│       ├── goals.md        # what you're working toward
│       ├── YYYY-MM-DD.md   # daily notes (root level)
│       ├── sessions/       # session memory dumps (auto-generated by hooks)
│       ├── logs/           # weekly/monthly summaries, archived daily notes
│       └── projects/       # planning docs, specs, research notes
│           └── {project-name}/
└── automations/            # scheduled workflows (optional)
```

### Why This Layout

Code and notes live in separate trees but are connected. Each project in `projects/` can have a matching directory in `vault/projects/` for planning docs, progress tracking, and research notes. The vault is the knowledge layer — it accumulates context over time. The code changes; the notes remember why.

**`workspace/` is NOT a git repo.** It's just a container. Each project and the vault have their own repos.

---

## Core Files

### tasks.md

The master task list. Single source of truth for what needs doing. When planning work or adding new tasks, update `tasks.md` first. In-session task tracking (like TodoWrite) is secondary and doesn't persist — it should mirror tasks.md, not replace it.

### goals.md

High-level goals — what you're working toward this quarter/year. Reference this when prioritizing tasks or deciding what to work on.

### Daily Notes (YYYY-MM-DD.md)

Daily notes live at vault root as `YYYY-MM-DD.md`. Log activities with timestamps:

```
- [HH:MM] Descriptive title #tags
	- specific detail (tab indented)
	- another detail
```

Always use `date` command before timestamps. Never guess times.

### Project Ledgers

Each project has a `ledger.md` at its root — the fastest handoff file for any agent resuming work. It captures what was done, what's blocked, what's next.

**Read it at session start. Update it before session end or context compaction.**

Append-only, newest entries at top. This is how context persists across sessions for a specific project. The ledger lives in the project repo, not the vault — it's about that project and travels with it.

### Logs

`vault/logs/` contains:
- Archived daily notes (moved here periodically to keep vault root clean)
- Weekly summaries — end-of-week review of what shipped, what stalled, what's next
- Monthly summaries — broader patterns, goal progress, priority shifts

## Working Protocols

### Finding Project Context

When asked to work on a project:
1. Read `projects/{project-name}/ledger.md` for recent progress and current focus
2. Read `projects/{project-name}/docs/plan.md` for direction
3. Read `projects/{project-name}/docs/resources.md` for reference material
4. Check `vault/tasks.md` for current priorities across all projects
5. Check `vault/projects/{project-name}/` for planning docs and specs (if they exist)

### Session Memory

Conversation history is automatically dumped to `vault/sessions/{session_id}.md` before each context compaction. This provides persistent memory across compactions within a session.

**To recover context after compaction**: Read the session file at `vault/sessions/{session_id}.md`. Find recent sessions by listing the directory and checking modification times.

This requires the `agent-memory` plugin (or manually installing the hooks from agent-utils).

### Memory Architecture

Four layers of persistent memory, each serving a different purpose:

| Layer | Location | Purpose | Loaded |
|---|---|---|---|
| **Auto-memory** | `~/.claude/projects/.../memory/MEMORY.md` | Stable facts, conventions, project state | Auto-loaded |
| **Session dumps** | `vault/sessions/{id}.md` | Full conversation recovery after compaction | On demand |
| **Project ledgers** | `projects/{name}/ledger.md` | Per-project session handoff | On demand |
| **Vault** | `vault/` (tasks, notes, daily logs) | Long-term knowledge and planning | On demand |

Auto-memory stores stable facts. Ledgers store chronological progress. Don't duplicate between them.

### Task Tracking

`vault/tasks.md` is the single source of truth. Format tasks however works for you — the important thing is that there's one place to look and it stays current.

### Note-Taking

Default: write to vault root. Let notes accumulate as loose files, then organize in batches. Don't over-categorize upfront.

### Project Kickoff

When starting a new project:
1. Create project directory in `projects/` with `CLAUDE.md`, `ledger.md`, `README.md`, `docs/`
2. Initialize `ledger.md` with product thesis and current focus
3. Add the project to `vault/tasks.md`
4. Optionally create `vault/projects/{project-name}/` for planning docs and specs

## Setup

1. Create the workspace structure:
   ```bash
   mkdir -p workspace/{projects,notes/vault/{sessions,logs,projects},automations}
   cp this-file workspace/CLAUDE.md
   touch workspace/notes/vault/{tasks,goals}.md
   ```

2. Initialize the vault as a git repo:
   ```bash
   cd workspace/notes/vault
   git init
   ```

3. Install memory hooks (optional but recommended):
   ```bash
   # Via plugin
   claude plugin add path/to/agent-utils/plugins/agent-memory

   # Set session dump location
   export SESSIONS_DIR=~/path/to/workspace/notes/vault/sessions
   ```

4. Index vault for search (optional):
   ```bash
   bun install -g @tobilu/qmd
   qmd index workspace/notes/vault
   ```

Obsidian is recommended for the vault but any markdown directory works. The key is that it's git-tracked and your agent can read/write to it.
